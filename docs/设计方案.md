# 实时行情监控与规则告警系统设计方案

## 1. 背景与目标

- 为黄金与股票等标的提供多源实时/准实时价格监控。
- 支持以“规则”为单位配置触发条件与通知方式，如：
  - 当价格到达 1140 元时发送通知；
  - 当价格到达 1160 元时再次发送通知；
  - 通知方式支持：飞书/企微/钉钉 Webhook，支持自定义 Webhook 地址。
- 提供可视化管理后台：维护标的、规则、Webhook。

## 2. 核心功能

- 标的管理：启停黄金、股票等监控对象（Instrument）。
- 规则管理：以标的为维度配置多个触发规则，支持多种条件与生效窗口。
- 告警投递：根据规则触发通过平台适配器（飞书/企微/钉钉）发送消息。

## 3. 概念与术语

- Instrument：监控标的（如 XAU、某 A 股、ETF）。
- Price Tick：标准化的价格事件（时间戳、买一/卖一/中间价、来源）。
- Rule：规则，定义触发条件（如到达/突破/回落/区间/涨跌幅）与通知动作。
- WebhookConfig：通知渠道配置（平台类型+URL+签名/密钥+模板参数）。

## 4. 典型用例

1. 内置工行积存金的标的（根据某个接口添加的适配器）。
2. 创建一条规则，当工行积存金的价格到达 1140 元时触发规则，记录触发时间防止此规则1小时内重复触发。
3. 定时任务：每分钟进行标的价格轮询，更新最新新价格，匹配触发规则。
4. 触发规则时发送通知：使用飞书/企微/钉钉 Webhook 发送消息，包含价格、时间戳、触发规则等信息。

## 5. 系统架构（极简版）

### 5.1 组件划分

- **核心服务（Monolithic Worker）**
  - **行情采集**：内置多个硬编码的行情源适配器（内存实现），按需轮询。
  - **规则引擎**：直接在采集流中匹配规则，无独立队列。
  - **告警投递**：匹配成功后直接调用 Webhook 发送。
- **存储**
  - **D1 数据库**：仅一张 `rules` 表，存储规则配置与当前状态（上次触发时间等）。
  - **KV / Secrets**：存储 Webhook 地址、API 密钥等敏感配置。

### 5.2 数据流

```
定时触发 (Cron)
    ↓
[ 采集 Worker ]
    ├─ 1. 内存执行适配器获取最新价 (Price Tick)
    ├─ 2. 从 D1 (rules 表) 加载活跃规则
    ├─ 3. 内存匹配规则 (Rule Engine)
    │     ├─ 满足条件?
    │     ├─ 冷却中? (检查 rules 表中的 state 字段)
    │     └─ 触发!
    ├─ 4. 发送 Webhook (Feishu/WeCom/DingTalk)
    └─ 5. 更新 D1 (回写 state 字段: lastFiredAt 等)
```

**说明：**

- **无状态设计**：鉴于 Cloudflare Workers 的无状态特性，不依赖内存缓存。
- **直接读库**：每次 Cron 任务直接查询 D1 获取规则列表。对于个人项目（规则数 < 100），D1 的读性能完全足够。
- **状态持久化**：规则触发后的状态直接更新回 D1 的 `rules` 表。

## 6. 数据源与采集策略

- 接入方式
  - HTTP 轮询：固定周期拉取最新价，适合免费或限流 API。
  - WebSocket/流：实时推送，适合交易所或第三方实时源。
  - 混合：主用 WS，兜底轮询。
- 标准化
  - Tick: `{ instrumentId, price, ts, source, bid?, ask?, mid? }`
  - 统一时区（UTC），统一精度（decimal/string + 精度字段）。
- 可靠性
  - 多源合并：优先级/时间戳择优；源异常自动降级。
  - 防抖：同毫秒内重复 Tick 合并；大幅波动优先处理。

## 7. 规则引擎设计

### 7.1 规则模型

```
Rule {
  id: string
  instrumentId: string
  name: string
  type: 'touch' | 'cross_up' | 'cross_down' | 'range' | 'pct_change'
  params: {
    target?: number            // 目标价（到达/突破）
    lower?: number, upper?: number  // 区间
    pct?: number               // 百分比（如 2%）
    windowMs?: number          // 计算窗口（涨跌幅/区间）
    holdMs?: number            // 触发需保持时间（如连续 3 秒）
  }
  // 通知时间
  notifyTime: number
  active: boolean
  schedule?: { tz: string, days?: number[], start?: string, end?: string } // 交易时段
  createdAt, updatedAt
}
```

类型说明：

- touch（到达）：价格首次进入 [target - ε, target + ε] 触发；默认 ε=0（可配置）。
- cross_up（上涨）：从小于 target 到大于等于 target 的边界穿越触发。
- cross_down（下跌）：反向穿越触发。
- range（区间）：进入/离开某区间触发，可拆分为两条子规则。
- pct_change（涨跌幅）：在 windowMs 内相对上一参考价涨跌超过阈值触发。

### 7.2 状态机与去抖动

- 为每条规则维护 State：
  - lastValue：上次价格；
  - lastFiredAt：上次告警时间；
  - inWindow：是否处于触发条件内；
  - cooldownUntil：抑制期结束时间；
  - refPrice：参考价（用于 pct_change）。
- 触发判定：
  - 满足条件且不在 cooldown；
  - 若 holdMs>0，需在持有窗口内连续满足后触发；
  - suppressUntilReset 时，只在从“不满足→满足”边沿触发一次，直到离开条件后复位。

### 7.3 示例（满足用户描述）

- 规则 A：触发条件为 touch(target=1140)，channels=[飞书/企微/钉钉 中任意配置]。
- 规则 B：触发条件为 touch(target=1160)，channels 同上。
- 两条规则相互独立，均有各自的去抖动与冷却设置。

## 8. 通知与 Webhook 适配

### 8.1 统一接口

```
Notifier.send({
  channel: 'feishu' | 'wecom' | 'dingtalk',
  webhookUrl: string,
  secret?: string,     // 某些平台需要签名
  title: string,
  text: string,
  card?: object,       // 平台卡片内容（可选）
  dedupeKey?: string,  // 幂等去重键
})
```

### 8.2 消息模板（示例）

- 标准文本：
  - 标的：{instrumentName}
  - 价格：{price}
  - 规则：{ruleName}（{type}/{target}）
  - 时间：{ts}
  - 链接：{detailUrl}

### 8.3 平台差异

- 飞书（Feishu）：支持文本、卡片消息；部分需要签名；机器人 Webhook。
- 企微（WeCom）：文本/Markdown/卡片；支持关键字/签名。
- 钉钉（DingTalk）：文本/Markdown/ActionCard；需 timestamp+sign。

统一通过适配器处理签名与 Content-Type，并在失败时返回平台错误码。

## 9. 存储设计

### 9.1 D1 数据库 (rules 表)

```sql
CREATE TABLE rules (
  id TEXT PRIMARY KEY,          -- UUID
  instrument_id TEXT NOT NULL,  -- 标的 ID
  name TEXT NOT NULL,           -- 规则名称
  type TEXT NOT NULL,           -- 规则类型: touch, cross_up, etc.
  params JSON NOT NULL,         -- 规则参数: {target, lower, upper, windowMs}
  notify JSON NOT NULL,         -- 通知配置: {channels: ["feishu_default"], throttleMs: 3600000}
  state JSON,                   -- 运行时状态: {lastFiredAt, lastValue, cooldownUntil}
  active BOOLEAN DEFAULT TRUE,
  created_at INTEGER DEFAULT (unixepoch()),
  updated_at INTEGER DEFAULT (unixepoch())
);
```

### 9.2 KV 存储设计 (Binding: KV_CONFIG)

采用扁平化 Key-Value 存储，直接保存字符串值，不使用 JSON。

| 配置项           | Key                | Value 示例                        | 说明               |
| :--------------- | :----------------- | :-------------------------------- | :----------------- |
| **管理员用户名** | `ADMIN_USER`       | `admin`                           | 管理后台登录用户名 |
| **管理员密码**   | `ADMIN_PASS`       | `password123`                     | 管理后台登录密码   |
| **飞书 Webhook** | `WEBHOOK_FEISHU`   | `https://open.feishu.cn/...`      | 完整 Webhook URL   |
| **企微 Webhook** | `WEBHOOK_WECOM`    | `https://qyapi.weixin.qq.com/...` | 完整 Webhook URL   |
| **钉钉 Webhook** | `WEBHOOK_DINGTALK` | `https://oapi.dingtalk.com/...`   | 完整 Webhook URL   |

**关联逻辑：**

1. 规则表 `notify.channels` 存储简单的标识符，如 `["feishu", "wecom"]`。
2. 代码运行时根据标识符映射到对应的 KV Key（如 `feishu` -> `WEBHOOK_FEISHU`）读取 URL。

## 11. 管理控制台（前端）

- 信息架构与导航
  - 顶部/侧边导航：仪表盘、标的、实时行情、规则、Webhook。
  - 全局搜索：按标的/规则名/ID 搜索并快速跳转到详情。
  - 全局状态：环境标识（staging/prod）、队列 backlog 指示灯、版本信息。

- 页面清单与功能细节 登录页（Login）
  - 路由：/login（仅在启用“会话 Cookie 模式”时生效；默认 Basic Auth 仍可作为回退方式，见第 19 章）
  - 目标：统一入口登录管理台，改善 Basic 原生弹窗体验；校验成功后设置短期会话 Cookie 并跳转 /admin
  - 表单：
    - 用户名（必填）
    - 密码（必填）
    - 记住登录（可选，启用时将 Cookie Max-Age 设为 7 天；未勾选则为会话期）
  - 交互：
    - 支持 Enter 提交；提交中按钮禁用并显示加载；提交失败聚焦错误提示
    - 错误类型区分：401 凭证错误、429 尝试过多（显示剩余等待时间）、其他网络错误
    - 成功后跳转：优先使用 URL 中的 redirect 参数（同域校验），否则跳转 /admin
  - 安全：
    - 后端采用常数时间比较；不记录明文凭证与 Authorization 头
    - 返回 Set-Cookie: goldwatch_sid=<签名值>; HttpOnly; Secure; Path=/; SameSite=Lax; Max-Age=<可选>
    - CSRF：登录接口使用双提交 Cookie 或者基于 SameSite=Lax；受保护写操作的 API 再加自定义 X-CSRF-Token（后续实现）
    - 失败限速：基于 IP/指纹的尝试计数与退避（429），避免暴力猜测
  - API：
    - POST /api/auth/login
      - 请求：{ username, password, remember?: boolean } 或 Authorization: Basic
      - 成功：200 + Set-Cookie（金签名会话）；body { ok: true }
      - 失败：401（凭证错误），429（超过尝试上限），5xx（服务错误）
    - POST /api/auth/logout
      - 行为：清除 goldwatch_sid；返回 204
  - 可访问性：
    - label/aria-describedby、aria-invalid、aria-live=polite 输出错误
    - 屏幕阅读器顺序与视觉顺序一致；焦点环可见
  - 兼容与回退：
    - 若浏览器阻拦第三方 Cookie 或 Set-Cookie 失败，页面提示改用 Basic Auth（示例 curl -u）或在同浏览器会话中访问受保护资源触发原生弹窗
  - 文案：固定简体中文，不做国际化与多语言
  1. 仪表盘（Dashboard）
     - KPI 卡片：监控标的数、启用规则数、近 24h 告警数、投递成功率。
     - 趋势图：告警趋势、投递成功率曲线、采集延迟分布（分钟级）。
     - 最近事件：最新 10 条 Alert 与 Delivery（可点击跳转详情）。
     - 系统状态：定时采集最近运行时间、队列 backlog、失败率。

  2) 标的管理（Instruments）

     - 操作：新建/编辑/复制/删除/启用/停用、批量启停、批量删除。

         - 试发：/api/rules/:id/test（以当前价或指定价模拟一次触发，不写入状态）；

## 12. 部署方案与选型（极简单 Worker）

- **运行平台**：Cloudflare Workers (Standard) + D1 (Beta) + KV
- **调度**：Cron Triggers（每分钟运行一次，执行采集+匹配）
- **成本**：
  - Workers: 免费版 100k 请求/天（个人完全够用）。
  - D1: 免费版 500万读/月（每分钟 1 次 _ 30 天 _ 1440 分钟 = 4.3万次查询，远低于限额）。
  - KV: 免费版 10万读/天。

## 13. 安全与合规（个人版）

- Webhook URL/Secret 加密存储，不在日志中输出明文；仅在发送时解密至内存。
- 出站请求限制与 allowlist；防止 SSRF。
- 请求签名与验签（对外开放 API 时）。

## 14. 性能与可靠性

- 目标：秒级触达；从 Tick 到投递 < 1s（取决于数据源与平台限流）。
- 限流：对外投递按渠道与域名限速，失败退避。
- 缓存：最新价存内存。
- 灰度：规则变更热更新，先对单标的生效，逐步全量。

## 15. 可观测性

- 告警：系统异常（连续失败、延迟高于阈值）上报至维护群。

## 18. 规则启停（Enable/Disable）

- 状态含义
  - active=true：规则参与评估与触发；
  - active=false：规则被跳过，不再触发新告警；历史状态保留；
  - 可选：停用时是否清理 state（cooldown、inWindow、refPrice）由参数控制。

- 引擎行为
  - 评估入口先判断 active；为 false 直接返回；
  - 停用→启用时：可选择“保留状态”或“复位状态”，避免启用瞬间误触发。

- API（新增）
  - POST /api/rules/:id/enable
    - body：{ resetState?: boolean }（默认 false）
    - 行为：将 active 置 true；若 resetState=true 则清理 rule_states 对应字段
  - POST /api/rules/:id/disable
    - body：{ clearCooldown?: boolean }（默认 true）
    - 行为：将 active 置 false；可选清理冷却窗口，保留最近值与参考价用于审计
  - POST /api/rules/batch-toggle
    - body：{ ids: string[], active: boolean, resetState?: boolean }
    - 行为：批量启停，适合交易时段或维护窗口操作

- 存储变更
  - rules.active：已有字段直接生效；
  - rule_states：新增字段可选 reset 标记（如 resetAt、resetBy）。

- 管理控制台（UI）
  - 在规则列表与详情页提供“启用/停用”切换；
  - 停用时弹出二次确认，可选是否清理状态/冷却；
  - 支持批量选择后统一启停；
  - 操作记录写入审计日志（操作人、时间、策略）。

- 使用场景
  - 行情极端波动期间临时停用部分规则；
  - 交易时段切换时按计划批量启用/停用；
  - 调整参数前先停用，验证后再启用，确保无误触发。

## 19. 用户登录认证（简化方案：环境变量 Basic Auth）

- 目标与范围
  - 本项目作为内部自用系统，不引入用户数据库与注册流程；
  - 部署时通过KV指定单一管理员用户名与密码；
  - 管理控制台（/admin）与受保护 API（/api/**，除 /api/public/**）均需认证。

- 用户名+密码
  - 生产环境
    - ADMIN_USER：管理员用户名（通过 KV 获取）
    - ADMIN_PASS：管理员密码（通过 KV 获取）
  - 开发环境
    - ADMIN_USER：默认值 "admin"
    - ADMIN_PASS：默认值 "password"

- 认证方式
  - 登陆后下发 Token 存储在 localStorage 中；
  - 后续请求在 Header 中携带 Token 进行认证；
  - 过期或无效 Token 时返回 401 并重新触发登陆流程。

- 路由与中间件策略
  - 受保护资源：
    - GET/POST/PATCH/DELETE /api/**（除 /api/public/**）
    - GET /admin/\*\*（管理控制台静态与数据接口）
  - 放行资源：
    - /api/public/\*\*（若将来提供无需认证的公开只读接口）
    - 静态首页或产品说明页（可选）
  - 在 Worker 入口添加认证中间件，对受保护路径进行 Basic 校验。

- API 行为规范（受保护接口）
  - 未认证或认证失败：返回 401 + WWW-Authenticate；
  - 不返回具体是哪项凭证错误；不回显输入；
  - 日志仅记录失败计数与来源（脱敏），不记录 Authorization 内容。
